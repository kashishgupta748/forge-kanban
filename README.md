## 📋 Project Overview

**Forge Kanban** is a production-ready project management application in the spirit of Trello and Linear. It features drag-and-drop boards, card management, team collaboration, and real-time activity tracking.

What makes Forge Kanban unique is *how it was built*: every line of code, every migration, every React component, and every deployment config was generated by a two-agent AI system operating autonomously over Slack. A human provides high-level direction; the agents plan, build, test, and deploy without further hand-holding.

**Key value proposition:**
- Demonstrates a realistic, production-grade human-in-the-loop agent workflow
- Shows intelligent model routing (planning model vs. coding model)
- Proves persistent cross-session memory enables consistent, coherent codebases
- Full-stack application with zero manual code written by the human operator

## 🔄 Agent Workflow

The Human→Hermes→OpenClaw pipeline follows a structured loop:

### Step-by-Step

```
1. HUMAN sends intent via Slack
   └─ "Add overdue card highlighting to the board"

2. HERMES receives & analyzes
   ├─ Recalls memory: "Cards have due_date field (nullable timestamp)"
   ├─ Recalls memory: "Overdue color convention: red-500 (#EF4444)"
   ├─ Recalls memory: "Card component at resources/js/components/Card.tsx"
   └─ Decomposes into tasks:
       [T-01] Add isOverdue computed prop to Card model
       [T-02] Include is_overdue in CardResource JSON
       [T-03] Update Card.tsx to show red border when overdue
       [T-04] Add overdue badge to CardModal.tsx

3. HERMES dispatches T-01 to OPENCLAW
   └─ JSON task: { type: "code", file: "app/Models/Card.php", instruction: "..." }

4. OPENCLAW executes T-01
   ├─ Reads existing Card.php
   ├─ Generates updated file with isOverdue() accessor
   ├─ Writes file to disk
   └─ Reports: { status: "done", file: "app/Models/Card.php", lines: 87 }

5. HERMES receives completion, dispatches T-02
   [... loop continues for T-03, T-04 ...]

6. HERMES runs status report (SKILL: status-report)
   └─ Posts to Slack: "✅ Overdue highlighting complete. Cards past due_date
      now show red border + 🔴 badge. No decisions needed."

7. HUMAN reviews and provides next direction
```

## 🚀 Installation

### Prerequisites

- PHP 8.3+
- Composer 2.x
- Node.js 20+
- npm 10+

### Backend Setup

```bash
# 1. Clone the repository
git clone https://github.com/your-username/forge-kanban.git
cd forge-kanban

# 2. Install PHP dependencies
composer install

# 3. Configure environment
cp .env.example .env
php artisan key:generate

# 4. Create SQLite database
touch database/database.sqlite

# 5. Run migrations and seed demo data
php artisan migrate --seed

# 6. Start the development server
php artisan serve
# API available at http://localhost:8000
```

### Backend — Render

1. Create a new **Web Service** at [render.com](https://render.com)
2. Connect your GitHub repository
3. Use these settings:
   - **Environment**: PHP
   - **Region**: Oregon (US West)
   - **Plan**: Free
   - **Build Command**: See `render.yaml`
   - **Start Command**: `php artisan serve --host=0.0.0.0 --port=$PORT`
4. Set these environment variables in Render dashboard:

   | Variable | Value |
   |----------|-------|
   | `APP_ENV` | `production` |
   | `APP_KEY` | *(generate with `php artisan key:generate --show`)* |
   | `APP_URL` | `https://forge-kanban-api.onrender.com` |
   | `FRONTEND_URL` | `https://forge-kanban.vercel.app` |
   | `DB_CONNECTION` | `sqlite` |
   | `SANCTUM_STATEFUL_DOMAINS` | `forge-kanban.vercel.app` |

> **Note:** Render free-tier services spin down after 15 minutes of inactivity. Use a cron ping service (e.g., UptimeRobot) to keep the backend warm.

## 🧠 Model Routing Rationale

The two-agent system uses different AI models for different roles, chosen deliberately:

### Hermes (Brain) — Gemini 2.5 Flash

| Criterion | Rationale |
|-----------|-----------|
| **Task type** | Planning, decomposition, reasoning, memory management — language tasks requiring broad understanding |
| **Context window** | 1M token context — can hold entire codebase in context for architectural decisions |
| **Reasoning quality** | Gemini 2.5 Flash's thinking mode excels at multi-step planning and constraint reasoning |
| **Cost** | Free tier via Google AI Studio for development; minimal cost in production |
| **Latency** | ~1-3s for planning tasks — acceptable since Hermes runs asynchronously |

**Fallback:** Groq `llama3-70b-8192` — used when Gemini API is rate-limited or unavailable. Groq's inference speed (500+ tok/s) makes it an excellent fallback with negligible latency penalty.

### OpenClaw (Hands) — qwen2.5-coder:32b via Ollama

| Criterion | Rationale |
|-----------|-----------|
| **Task type** | Code generation, file writing, command execution — specialized coding tasks |
| **Model specialization** | qwen2.5-coder is trained specifically on code; outperforms general models on PHP/TypeScript/React generation |
| **Cost** | Zero — runs locally via Ollama. No API costs for potentially hundreds of code generation calls |
| **Privacy** | Code never leaves the local machine during generation |
| **Context** | 32B parameter model provides strong quality while running on consumer hardware (32GB RAM) |

**Fallback:** Groq `mixtral-8x7b-32768` — used when local Ollama is unavailable (e.g., during cloud runs). Mixtral's MoE architecture gives strong code quality with fast inference.

### Routing Decision Tree

```
Incoming Task
     │
     ├── Is it planning/reasoning/memory?
     │        └── YES → Hermes (Gemini 2.5 Flash)
     │                   └── Rate limited? → Groq llama3-70b
     │
     └── Is it code/commands/files?
              └── YES → OpenClaw (qwen2.5-coder:32b local)
                         └── Ollama down? → Groq mixtral
```

3. Subscribe to `message.im` and `message.channels` events
4. Add bot scopes: `chat:write`, `channels:read`, `channels:history`
5. Install to your workspace and copy the **Bot Token** + **Webhook URL**

## 🤝 Contributing

Contributions are welcome! Please follow these guidelines:

1. **Fork** the repository and create a feature branch:
   ```bash
   git checkout -b feature/your-feature-name
   ```

2. **Follow code style**:
   - PHP: PSR-12 standard (enforced by Laravel Pint)
   - TypeScript: ESLint + Prettier (`.eslintrc` and `.prettierrc` included)

3. **Write tests** for new features:
   - Backend: PHPUnit feature tests in `tests/Feature/`
   - Frontend: Vitest unit tests in `src/__tests__/`

4. **Commit with conventional commits**:
   ```
   feat: add checklist support to cards
   fix: correct overdue date comparison timezone issue
   docs: update API reference with /checklist endpoints
   ```

5. **Open a Pull Request** — describe what changed and why, include screenshots for UI changes.

6. **CI must pass** — GitHub Actions runs tests, linting, and type-checking on every PR.
